“金丝雀发布”（Canary Release）是一种**渐进式发布策略**，用于在生产环境中**降低上线风险**。

---

## 🐤 什么是金丝雀发布？

名字来源于矿井里的“金丝雀”（canary）——矿工会先让金丝雀进入矿井，如果有毒气，金丝雀会先出问题，从而保护人类。

👉 在软件里也是同样逻辑：

> **先让一小部分用户使用新版本，如果没问题，再逐步扩大范围。**

---

## 🚀 发布流程（典型步骤）

1. 发布新版本（v2）到一小部分实例（比如 5%）
2. 只让一小部分用户流量进入 v2
3. 监控关键指标：
   - 错误率（error rate）
   - 延迟（latency）
   - CPU / 内存

4. 如果一切正常：
   - 扩大流量（20% → 50% → 100%）

5. 如果有问题：
   - **快速回滚到旧版本（v1）**

---

## 📊 举个例子（真实场景）

比如你有一个 API 服务：

- v1：老版本（100%流量）
- v2：新版本

发布方式：

| 阶段   | v1流量 | v2流量 |
| ------ | ------ | ------ |
| 初始   | 100%   | 0%     |
| 金丝雀 | 95%    | 5%     |
| 扩大   | 50%    | 50%    |
| 全量   | 0%     | 100%   |

---

## 🆚 和其他发布方式对比

| 发布方式               | 特点             | 风险           |
| ---------------------- | ---------------- | -------------- |
| 蓝绿发布（Blue-Green） | 两套环境切换     | 切换瞬间风险大 |
| 滚动发布（Rolling）    | 分批替换实例     | 无精细流量控制 |
| **金丝雀发布**         | 按“用户流量”渐进 | ✅ 最安全      |

---

## 🧠 核心技术点

实现金丝雀发布通常依赖：

### 1. 流量控制（核心）

- 按比例分流（5%、10%）
- 按用户分流（userId hash）
- 按Header/Cookie

👉 常用工具：

- Kubernetes + Istio
- NGINX
- Envoy

---

### 2. 指标监控

- Prometheus + Grafana
- 日志 + tracing

---

### 3. 自动化发布

- CI/CD pipeline
- 自动 rollback（如果 error rate 超阈值）

---

## ⚠️ 注意点（面试高频）

1. **如何选择金丝雀用户？**
   - 随机
   - 内部员工
   - 指定地区

2. **如何避免数据不一致？**
   - 数据库 schema 向前兼容
   - 双写（write to both versions）

3. **如何快速回滚？**
   - 流量直接切回 v1（不需要 redeploy）

---

## 💡 一句话总结

> 金丝雀发布 = **小流量试错 + 实时监控 + 可快速回滚**
